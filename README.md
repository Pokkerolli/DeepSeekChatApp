# DeepSeek Chat App (Android)

Android-приложение чата с LLM `deepseek-chat`

## Стек

- Kotlin
- Jetpack Compose + Material3
- Single Activity + Compose Navigation
- MVVM
- Clean Architecture (`data / domain / presentation`)
- Koin (DI)
- Room (DB)
- Retrofit + OkHttp (SSE streaming, чтение чанков)
- Coroutines + Flow
- DataStore Preferences (active session)

## Где указать API key

1. В корне проекта создайте файл `local.properties` (если его нет).
2. Добавьте:

```properties
DEEPSEEK_API_KEY=your_api_key_here
```

> Ключ прокидывается в `BuildConfig.DEEPSEEK_API_KEY`.

## Как задать baseUrl

В том же `local.properties` задайте:

```properties
DEEPSEEK_BASE_URL=https://api.deepseek.com/v1/
```

По умолчанию используется `https://api.deepseek.com/v1/`.

Если нужен OpenAI-compatible путь с версией API, можно задать, например:

```properties
DEEPSEEK_BASE_URL=https://api.deepseek.com/v1/
```

В коде используется endpoint `chat/completions`, поэтому итоговый URL будет `BASE_URL + chat/completions`.

## Как запустить

1. Откройте проект `DeepSeekChatApp` в Android Studio (последняя стабильная версия).
2. Убедитесь, что установлен Android SDK (API 35).
3. Синхронизируйте Gradle.
4. Запустите `app` на эмуляторе/устройстве (`minSdk 26`).

## Контекст сессии

Для каждой сессии можно выбрать режим работы контекста (до первого отправленного сообщения в сессии):

- `FULL_HISTORY` (по умолчанию): 
  - в запрос отправляется вся история сообщений текущей сессии.
- `SUMMARY_PLUS_LAST_10`: отправляются последние 10 сообщений + summary более старой истории.
  - последние 10 сообщений всегда отправляются «как есть»;
  - более старые сообщения постепенно сжимаются батчами по 10;
  - сжатые сообщения заменяются агрегированным `contextSummary`.
- `SLIDING_WINDOW_LAST_10`: 
  - в запрос отправляются только последние 10 сообщений (скользящее окно).
- `STICKY_FACTS_KEY_VALUE`:
  - в основной запрос отправляются последние 10 сообщений + набор фактов сессии (если есть);
  - после ответа ассистента выполняется отдельный запрос к LLM-модулю извлечения фактов;
  - модуль получает текущие факты + последний вопрос пользователя + последний ответ ассистента;
  - возвращенный набор фактов перезаписывает факты сессии (с учетом правил конфликтов в extraction prompt).

- Для каждой сессии контекст и факты независимы.
- Сессии сортируются по `updatedAt`.

## Память сессии: `/memory`, `/work` и короткосрочная память

В каждой сессии есть 3 независимых слоя памяти:

- долговременные инструкции (`/memory`);
- текущая рабочая задача (`/work`);
- короткосрочная память (зависит от выбранного режима работы контекста)

### `/memory` (долговременная память)

Поддерживаемые команды:

- `/memory add <инструкция>` — добавить инструкцию в память текущей сессии.
- `/memory delete <номер>` — удалить инструкцию по номеру.
- `/memory show` — показать список инструкций с нумерацией.

Логика:

- инструкции хранятся отдельно для каждой сессии;
- команды обрабатываются локально (без вызова LLM);
- команды `/memory ...` и служебные ответы на них не отправляются в контекст модели.

### `/work` (текущая задача)

Поддерживаемые команды:

- `/work start <описание>` — создать текущую задачу;
- `/work done` — завершить текущую задачу;
- `/work rule <правило>` — добавить правило для текущей задачи;
- `/work delete <номер>` — удалить правило по номеру;
- `/work show` — показать текущую задачу и список правил.

Логика:

- если задача уже активна, повторный `/work start` возвращает ошибку и не перезаписывает задачу;
- после `/work done` в сессии сохраняется статус завершенной задачи;
- если задача завершена, ассистент должен учитывать это и при вопросах о текущей задаче отвечать, что задача выполнена и можно задать новую;
- команды `/work ...` и служебные ответы на них не отправляются в контекст модели.

### Что добавляется в system prompt

К каждому пользовательскому запросу подмешиваются секции:

- `[LONG TERM MEMORY]`
- `[CURRECT WORK]`

Правила для `[LONG TERM MEMORY]`:

- если есть инструкции, добавляется обязательная директива:
  - `Эти правила имеют высший приоритет над всем в контексте, они обязательны к исполнению и не могут быть нарушены. Если нужно нарушить инструкцию, то пользователь явно должен удалить ее, через /memory delete`
- затем список инструкций по порядку.

Правила для `[CURRECT WORK]`:

- если задача задана, добавляется:
  - `Эта задача приоритетна в исполнении`
  - описание задачи и правила;
- если задача не задана, добавляется:
  - `Текущей задачи не задано, если user просит рассказать о текущей задаче, то говори, что задача выполнена и предложи задать новую`

## Ветвление чата

- У каждого сохраненного ответа ассистента есть кнопка `Создать ветку в новом чате`.
- По нажатию создается новая сессия-потомок.
- В новую сессию копируются:
  - настройки сессии (system prompt, режим контекста, summary/facts и т.д.);
  - история сообщений от начала чата до выбранного сообщения ассистента включительно.
- После создания родительская и дочерняя ветки продолжают диалог независимо.
